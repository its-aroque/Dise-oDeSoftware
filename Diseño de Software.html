Diseño de Software

Mejores prácticas y principios fundamentales

¿Qué es el Diseño de Software?

El diseño de software es un proceso fundamental en el desarrollo de soluciones informáticas que permite materializar los requerimientos del cliente y establecer la calidad del proyecto.

Definición

Proceso mediante el cual un agente crea una especificación de un artefacto de software para cumplir objetivos, utilizando componentes primitivos y sujeto a restricciones.

Alcance

Actividad implicada en conceptualizar, enmarcar, implementar, poner en funcionamiento y modificar sistemas complejos.

Ubicación en el ciclo

Actividad que sigue a la especificación de requisitos y precede a la programación en un proceso de ingeniería de software.

Principios Generales de Diseño

  • Considerar enfoques alternativos: Evaluar diversas soluciones basadas en requisitos y recursos.
  • Trazabilidad al modelo de análisis: Asegurar que cada elemento del diseño pueda rastrearse hasta los requisitos.
  • No reinventar la rueda: Utilizar patrones de diseño existentes para agilizar el desarrollo.
  • Minimizar la "distancia intelectual": La estructura del diseño debe imitar la estructura del dominio del problema.
  • Uniformidad e integración: Definir reglas de estilo consistentes y interfaces bien definidas.
  • Estructurado para adaptarse al cambio: Permitir la evolución del software evitando reescrituras costosas.
  • Degradación suave: El software debe operar bajo estrés y terminar de forma elegante si es necesario.
  • Evaluación continua de la calidad: Evaluar la calidad durante la creación, no después.
  • Revisión para minimizar errores conceptuales: Enfocarse en solucionar omisiones, ambigüedades e incongruencias.

Conceptos Fundamentales de Diseño

Abstracción

Representar características esenciales sin incluir detalles de fondo.

Refinamiento

Descomponer funciones macroscópicas en instrucciones más detalladas.

Modularidad

Dividir el software en componentes independientes y bien definidos.

Ocultamiento de Información

Diseñar módulos para que la información interna sea inaccesible para otros módulos que no la necesiten.

Jerarquía

Organizar el control y el trabajo de arriba hacia abajo en la estructura del programa.

Principios SOLID para Diseño Orientado a Objetos

SRP

Principio de Responsabilidad Única: Una clase o módulo debe tener solo una razón para cambiar.

OCP

Principio Abierto/Cerrado: Las clases deben estar abiertas a la extensión, pero cerradas a la modificación.

LSP

Principio de Sustitución de Liskov: Las subclases deben ser sustituibles por sus clases base sin producir resultados extraños.

ISP

Principio de Segregación de Interfaces: Es preferible tener muchas interfaces específicas que una interfaz de propósito general.

DIP

Principio de Inversión de Dependencias: Las clases deben depender de interfaces o clases abstractas, no de implementaciones concretas.

Prácticas para Código Limpio y Arquitectura Escalable

Código legible

Dedicar tiempo a que el código se entienda fácilmente, como si fuera prosa bien escrita.

Nombres significativos

Usar nombres que revelen la intención, pronunciables, buscables y que eviten la desinformación.

Funciones pequeñas

Las funciones deben ser reducidas y realizar "una sola cosa" a un único nivel de abstracción.

Manejo de errores

Utilizar excepciones en lugar de códigos de retorno. El procesamiento de errores debe ser separado de la lógica principal.

Evitar duplicación (DRY)

Los duplicados son indicadores de oportunidades de abstracción fallidas.

Pruebas de unidad

El código de prueba es tan importante como el de producción. Debe seguir principios F.I.R.S.T.
